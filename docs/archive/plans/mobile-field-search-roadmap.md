# План развития мобильного сервиса поиска элементов металлоконструкций

**Дата:** 2026-06-04  
**Проект:** IFC Engine WASM Viewer  
**Целевой сценарий:** монтажник на стройке открывает ссылку на модель металлоконструкции, вводит марку элемента и сразу видит нужный элемент в 3D.

## 1. Решаемая задача

Сервис должен быть не просто мобильным IFC/FRAG viewer, а полевым инструментом для быстрого поиска монтажных элементов по значениям атрибутов.

Ключевой пользовательский сценарий:

1. Инженер или BIM-координатор отправляет монтажнику ссылку на модель.
2. Монтажник открывает ссылку на телефоне.
3. Модель загружается автоматически.
4. Монтажник открывает поиск.
5. Вводит марку элемента, например `Б1-24`, `КМ-12`, `Ф-203`, `COL-05`.
6. Система находит элемент, подсвечивает его и приближает камеру.
7. Монтажник понимает, где элемент находится в конструкции.

Главный критерий качества: от открытия ссылки до визуального нахождения элемента должно проходить минимум действий и времени.

## 2. Продуктовое позиционирование

Сервис следует развивать как мобильный монтажный помощник для металлоконструкций.

Не основной фокус:

- полноценная BIM-система;
- универсальный IFC-редактор;
- сложный просмотр всех свойств;
- desktop-first интерфейс.

Основной фокус:

- быстро открыть модель по ссылке;
- быстро найти элемент по марке или атрибуту;
- визуально выделить элемент;
- дать монтажнику короткую понятную карточку элемента;
- в дальнейшем — собирать монтажные статусы и замечания с площадки.

## 3. Принципы UX

1. **Одна главная задача на экране:** найти элемент.
2. **Мобильный интерфейс первым:** крупные кнопки, нижняя зона управления, минимум мелкого текста.
3. **Поиск без знания IFC:** монтажник вводит привычную марку, а не имя IFC-поля.
4. **Визуальный результат важнее таблицы:** найденный элемент должен быть подсвечен и приближен камерой.
5. **Работа по ссылкам:** ссылка может вести не только на модель, но и на конкретный элемент.
6. **Минимум действий:** открыть → найти → увидеть.

## 4. Этапы развития

### Этап 1. MVP: быстрый поиск по марке

Цель: закрыть основной сценарий монтажника.

Функции:

- Поиск по нескольким IFC/FRAG атрибутам:
  - `Name`;
  - `Tag`;
  - `Mark`;
  - `ObjectType`;
  - `PredefinedType`;
  - пользовательские property set поля, где может храниться марка.
- Нормализация запроса:
  - без учёта регистра;
  - пробелы, дефисы и подчёркивания приводятся к единому виду;
  - поддержка кириллицы и латиницы в марках.
- Если найден один элемент:
  - подсветить элемент;
  - приблизить камеру;
  - приглушить остальные элементы;
  - показать компактную карточку результата.
- Если найдено несколько элементов:
  - показать список совпадений;
  - по нажатию на результат — подсветить и приблизить.

Критерий готовности:

- Монтажник вводит марку и за 1–2 секунды видит нужный элемент на телефоне.

### Этап 2. Мобильный UX поиска

Цель: сделать поиск удобным одной рукой на стройке.

Функции:

- Большая строка поиска в нижней части экрана.
- Автофокус при открытии поиска.
- Кнопка очистки запроса.
- История последних поисков в рамках текущей модели.
- Список результатов с короткими полями:
  - марка;
  - тип элемента;
  - уровень / отметка;
  - оси / зона, если есть;
  - короткий GUID или внутренний ID.
- Кнопки рядом с результатом:
  - `Показать`;
  - `Изолировать`;
  - `Сбросить`.

Ограничения:

- Не выводить длинное дерево свойств по умолчанию.
- Не заставлять пользователя выбирать поле поиска.
- Не делать мелкие кнопки менее 48 px по высоте.

### Этап 3. Ссылки на конкретный элемент

Цель: инженер может отправить монтажнику не просто модель, а конкретную марку.

Формат ссылки:

```text
/ifc-engine-wasm/viewer/?fragment=<fragment_id>&mark=<element_mark>
```

Поведение:

1. Viewer открывается по ссылке.
2. Загружается FRAG-модель.
3. После загрузки автоматически выполняется поиск по `mark`.
4. Найденный элемент подсвечивается.
5. Камера приближается к элементу.
6. Открывается карточка найденного элемента.

Пример:

```text
/ifc-engine-wasm/viewer/?fragment=28a746175560477d86ffe2c63393535b&mark=Б1-24
```

Критерий готовности:

- При открытии ссылки монтажник сразу видит нужный элемент без ручного ввода.

### Этап 4. Серверный индекс атрибутов

Цель: ускорить поиск и снизить нагрузку на телефон.

При сохранении FRAG-модели на сервере нужно формировать JSON-индекс элементов.

Пример структуры:

```json
{
  "fragment_id": "28a746175560477d86ffe2c63393535b",
  "created_at": "2026-06-04T00:00:00Z",
  "items": [
    {
      "modelId": "model-1",
      "localId": 12345,
      "guid": "3DkExampleGuid",
      "mark": "Б1-24",
      "name": "Beam B1-24",
      "type": "IfcBeam",
      "level": "Отм. +6.000",
      "zone": "Ось А-Б / 1-3",
      "profile": "30Б1",
      "material": "С245",
      "searchText": "б1-24 beam ifcbeam отм +6.000 30б1 с245"
    }
  ]
}
```

Что индексировать:

- IFC GUID;
- local ID / express ID;
- имя элемента;
- марку;
- тип IFC;
- уровень / отметку;
- оси / зону;
- профиль;
- материал;
- массу и длину, если есть в атрибутах;
- нормализованный `searchText`.

Плюсы:

- быстрый поиск без полного обхода свойств в браузере;
- возможность искать до полной загрузки детальных свойств;
- стабильный результат на слабых телефонах;
- проще добавить fuzzy search и подсказки.

### Этап 5. Умный поиск

Цель: монтажник не обязан вводить марку идеально.

Функции:

- Поиск без учёта регистра.
- Нормализация вариантов:
  - `Б1-24`;
  - `Б1 24`;
  - `Б1_24`;
  - `б1-24`;
  - `B1-24`, если в проекте смешаны кириллица и латиница.
- Частичные совпадения.
- Подсказки по мере ввода.
- Fuzzy search при небольшой опечатке.
- Ранжирование результатов:
  1. точное совпадение марки;
  2. совпадение нормализованной марки;
  3. совпадение имени;
  4. совпадение прочих атрибутов;
  5. fuzzy match.

Пример:

- Запрос: `б12`
- Возможные результаты:
  - `Б1-2`;
  - `Б1-20`;
  - `Б12-1`.

### Этап 6. Карточка монтажного элемента

Цель: после нахождения элемента дать монтажнику минимум нужной информации.

Карточка должна показывать:

- марку;
- тип элемента: балка, колонна, связь, ферма, пластина и т.п.;
- IFC class;
- уровень / отметку;
- оси / зону;
- профиль;
- материал;
- длину;
- массу;
- GUID / ID.

Действия в карточке:

- `Изолировать`;
- `Показать в модели`;
- `Сбросить вид`;
- `Скопировать марку`;
- `Скопировать ссылку на элемент`.

Минимальная версия карточки:

- марка;
- тип;
- GUID / ID;
- кнопки `Изолировать` и `Сбросить`.

### Этап 7. Режим монтажа

Цель: превратить viewer в лёгкий полевой журнал монтажа.

Функции:

- Статус элемента:
  - `Не начат`;
  - `В монтаже`;
  - `Смонтирован`;
  - `Проблема`.
- Комментарий к элементу.
- Фото с площадки.
- Тип проблемы:
  - `не найден на площадке`;
  - `не совпадает отверстие`;
  - `повреждён`;
  - `не хватает детали`;
  - `другое`.
- Экспорт замечаний.

Это уже следующий продуктовый уровень. Его не нужно делать раньше стабильного поиска по марке.

### Этап 8. Роли и доступы

Цель: разделить публичный просмотр, монтажные действия и инженерные функции.

Типы ссылок:

- **Публичная read-only ссылка:** просмотр модели и поиск.
- **Ссылка монтажника:** поиск, карточка, статусы, комментарии, фото.
- **Ссылка инженера:** свойства, диагностика, загрузка/обновление моделей.
- **Временная ссылка:** доступ с ограничением по времени.
- **Ссылка на конкретный элемент:** модель + автоматический поиск марки.

Приоритет: сначала read-only и ссылка на конкретный элемент, потом роли.

## 6. Рекомендуемый следующий шаг

Следующей задачей стоит сделать:

> Поиск по марке элемента с автоприближением и изоляцией найденного элемента, плюс поддержка ссылки `?mark=...`.

Почему это первый приоритет:

- напрямую закрывает основной сценарий монтажника;
- использует уже существующую модель публичных ссылок;
- не требует сразу строить полноценную систему статусов;
- быстро даст проверяемую ценность на стройке.

## 7. Acceptance criteria для следующей реализации

Сценарий 1: ручной поиск

1. Открыть viewer по публичной ссылке с `fragment`.
2. Дождаться загрузки модели.
3. Открыть поиск.
4. Ввести марку элемента.
5. Нажать результат.
6. Проверить, что элемент подсвечен, камера приблизилась, остальные элементы не мешают.

Сценарий 2: ссылка на элемент

1. Открыть URL вида:

```text
/ifc-engine-wasm/viewer/?fragment=<fragment_id>&mark=<element_mark>
```

2. Проверить, что модель загрузилась.
3. Проверить, что поиск выполнился автоматически.
4. Проверить, что найденный элемент подсвечен и открыт в карточке.

Сценарий 3: несколько совпадений

1. Ввести неполную марку.
2. Проверить, что показан список результатов.
3. Выбрать один результат.
4. Проверить подсветку и приближение камеры.

Сценарий 4: ничего не найдено

1. Ввести несуществующую марку.
2. Проверить понятное сообщение: `Ничего не найдено`.
3. Интерфейс не должен зависать или сбрасывать модель.
